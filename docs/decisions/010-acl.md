# ADR 010 — Audience ACLs: derived from the path, enforced at the answer

**Status:** accepted · 2026-07-13

## Context

Everything ingested was visible to every client (gbrain's per-source scoping is all-or-nothing
over the corpus). For a real company that is the #1 adoption blocker: payroll, board minutes and
client financials cannot share one visibility scope with the product roadmap.

## Decision

Audience labels with the same shape as entity resolution — **who may see a document is a
property of where it lives**, mapped by ordered config, never by an LLM:

- **Resolution (clean, pure code)**: `CLEAN_ACL` points at a JSON of ordered rules
  (`path_prefix` / `path_contains` / `unit` / `entity_kind` → audiences; first match wins;
  explicit default). The resolved list lands on the page (`acl: [...]`), on every facts row
  (a number inherits its document's audience), and on dossiers as the **intersection** of their
  members' audiences — a rollup must never widen access to what it summarizes. A malformed
  config errors loudly: silently-open is the one failure mode an access-control file must not
  have. No config = open corpus (no `acl` field), fully backward compatible.
- **Enforcement (answer, pure code)**: one visibility rule everywhere — no ACL → visible to
  all; unrestricted client → sees everything; else intersect. Search drops out-of-scope pages
  (not annotates: an invisible page simply isn't there), `read_page` denies, `query_metrics`
  filters rows, and even entity *existence* is scoped. Audiences are a **deployment property**
  (`ANSWER_AUDIENCES`): one server instance = one ACL scope; multi-tenant = one instance per
  scope, exactly gbrain's per-client pattern.

## Consequences

- The eval proves the product behavior: the same question is answered for a sales-scoped client
  and **refused** for an engineering-scoped one, with the restricted page absent from its
  search results.
- Facts and dossiers can't leak what their pages restrict (inheritance + intersection).
- Deliberate limitation, documented: this maps *conventions* to audiences. Mirroring live
  per-file Drive permissions is a connector concern — the fetch sidecar already persists
  whatever metadata Drive returns, and would feed the same `acl` field.

## Alternatives rejected

- **Per-request multi-tenant auth in the answer server** — real OAuth belongs in front (or in
  gbrain); a static per-instance scope is honest, testable and composable with any ingress.
- **LLM-assigned sensitivity labels** — access control must be deterministic and auditable.
- **Filtering at ingestion** (separate corpora per audience) — N corpora to keep consistent;
  labels + one enforcement point beat data duplication.
